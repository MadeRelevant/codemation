# Custom Credential Types

Custom credential types are how you make credentials feel like part of the framework instead of a pile of environment variables.

## The shape of a credential type

A credential type includes three parts:

1. `definition`
2. `createSession`
3. `test`

The repository uses this pattern:

```ts
const myCredentialType = {
  definition: {
    typeId: "my-service.apiKey",
    displayName: "My Service API key",
    publicFields: [{ key: "baseUrl", label: "Base URL", type: "string", required: true }],
    secretFields: [{ key: "apiKey", label: "API key", type: "password", required: true }],
    supportedSourceKinds: ["db", "env", "code"] as const,
  },
  createSession: async (args) => ({
    baseUrl: String(args.publicConfig.baseUrl ?? ""),
    apiKey: String(args.material.apiKey ?? ""),
  }),
  test: async (args) => {
    if (!args.publicConfig.baseUrl || !args.material.apiKey) {
      return {
        status: "failing" as const,
        message: "Base URL and API key are required.",
        testedAt: new Date().toISOString(),
      };
    }
    return {
      status: "healthy" as const,
      testedAt: new Date().toISOString(),
    };
  },
} satisfies CredentialType<MyPublicConfig, MyMaterial, MySession>;
```

## Register the credential type

Register custom credential types from `codemation.config.ts`:

```ts
register(app: CodemationAppContext) {
  app.registerCredentialType(myCredentialType);
  app.discoverWorkflows("src/workflows");
}
```

That keeps credential registration next to the rest of your app composition.

## Real examples in this repo

The sample app registers:

- `azureFoundry.contentUnderstandingApiKey`
- `odoo.demo`

Those examples show:

- public fields such as `endpoint` or `baseUrl`
- secret fields such as `apiKey`
- typed sessions returned by `createSession`
- simple `test(...)` implementations

## Link credentials to nodes

Nodes do not receive credentials automatically. They request them through a slot key and accepted type IDs.

That means your node or supporting config should expose a requirement like:

```ts
{
  slotKey: "myService",
  label: "My Service",
  acceptedTypes: ["my-service.apiKey"],
}
```

At runtime, the slot is bound to a concrete credential instance.

## Full example: define a slot and use the credential in `execute()`

The most useful mental model is:

1. the **credential type** defines what can be configured
2. the **node config** defines which slot and type IDs it accepts
3. the **node implementation** resolves the typed session inside `execute()`

```ts
import type {
  CredentialRequirement,
  CredentialType,
  Item,
  Items,
  Node,
  NodeExecutionContext,
  NodeOutputs,
  RunnableNodeConfig,
  TypeToken,
} from "@codemation/core";
import { node } from "@codemation/core";

type MyServicePublicConfig = Readonly<{ baseUrl?: string }>;
type MyServiceMaterial = Readonly<{ apiKey?: string }>;
type MyServiceSession = Readonly<{ baseUrl: string; apiKey: string }>;

export const myServiceCredentialType = {
  definition: {
    typeId: "my-service.apiKey",
    displayName: "My Service API key",
    publicFields: [{ key: "baseUrl", label: "Base URL", type: "string", required: true }],
    secretFields: [{ key: "apiKey", label: "API key", type: "password", required: true }],
    supportedSourceKinds: ["db", "env", "code"] as const,
  },
  createSession: async (args) => ({
    baseUrl: String(args.publicConfig.baseUrl ?? ""),
    apiKey: String(args.material.apiKey ?? ""),
  }),
  test: async (args) => ({
    status:
      String(args.publicConfig.baseUrl ?? "").trim() !== "" && String(args.material.apiKey ?? "").trim() !== ""
        ? ("healthy" as const)
        : ("failing" as const),
    testedAt: new Date().toISOString(),
  }),
} satisfies CredentialType<MyServicePublicConfig, MyServiceMaterial, MyServiceSession>;

export class MyServiceLookup implements RunnableNodeConfig<Record<string, unknown>, Record<string, unknown>> {
  readonly kind = "node" as const;
  readonly type: TypeToken<unknown> = MyServiceLookupNode;

  constructor(
    public readonly name: string,
    public readonly cfg: Readonly<{ credentialSlotKey?: string }>,
    public readonly id?: string,
  ) {}

  getCredentialRequirements(): ReadonlyArray<CredentialRequirement> {
    return [
      {
        slotKey: this.cfg.credentialSlotKey ?? "myService",
        label: "My Service API key",
        acceptedTypes: ["my-service.apiKey"],
      },
    ];
  }
}

@node()
export class MyServiceLookupNode implements Node<MyServiceLookup> {
  kind = "node" as const;
  outputPorts = ["main"] as const;

  async execute(items: Items, ctx: NodeExecutionContext<MyServiceLookup>): Promise<NodeOutputs> {
    const session = await ctx.getCredential<MyServiceSession>(ctx.config.cfg.credentialSlotKey ?? "myService");
    const out: Item[] = [];

    for (const item of items) {
      out.push({
        ...item,
        json: {
          ...(typeof item.json === "object" && item.json !== null ? item.json : {}),
          myServiceBaseUrl: session.baseUrl,
        },
      });
    }

    return { main: out };
  }
}
```

## Register the credential type and use the node

Register the type from `codemation.config.ts`:

```ts
register(app: CodemationAppContext) {
  app.registerCredentialType(myServiceCredentialType);
  app.discoverWorkflows("src/workflows");
}
```

Then use the node in a workflow:

```ts
new MyServiceLookup("Lookup customer", { credentialSlotKey: "myService" });
```

In the UI and runtime, that slot can then be bound to a credential instance of type `my-service.apiKey`.

## When to create one

Create a custom credential type when:

- you own a custom node package
- you integrate with a non-standard SaaS or internal API
- you want users to configure credentials in a structured and testable way

## Built-in reference

If you want a production-quality reference implementation, look at the built-in `openai.apiKey` credential type provided by the host package.

It shows a clean pattern for:

- typed public and secret fields
- optional `baseUrl`
- session creation
- health testing
