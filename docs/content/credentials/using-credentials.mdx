# Use Credentials

Credentials in Codemation are typed runtime resources, not random environment variables sprinkled through workflow code.

At a high level:

- a **credential type** defines the schema and runtime session creation logic,
- a **credential instance** stores a concrete configured credential,
- a **workflow node** declares which credential type it accepts through a named slot.

## The built-in OpenAI credential

Codemation ships with a built-in credential type:

- `typeId`: `openai.apiKey`

It includes:

- a required secret field for the API key
- an optional public `baseUrl`
- support for `db`, `env`, and `code` source kinds

That makes it a good default for `AIAgent` workflows and OpenAI-compatible endpoints.

## How nodes request credentials

Nodes and supporting configs declare credential requirements using a slot key plus accepted credential type IDs.

For example, the built-in OpenAI chat model config declares a slot and accepts `openai.apiKey`.

That gives you a simple pattern:

1. register or use an available credential type
2. create and bind a credential instance in the runtime
3. reference the matching slot from your workflow configuration

## Real custom credential example

The sample app in this repository registers custom credential types for:

- Azure AI Content Understanding
- a demo Odoo API

That is a good reference if you want typed credentials for your own integrations.

## When to use custom credentials

Create your own credential type when:

- a node depends on a non-standard API key or token
- users need structured public fields such as `baseUrl`, `endpoint`, or tenant IDs
- you want a proper `test(...)` step instead of just checking for env vars

## Source kinds

Credential types can support one or more source kinds:

- `db`
- `env`
- `code`

That lets you choose whether credentials are stored in the application database, resolved from environment variables, or supplied directly by code.

## Best practices

- prefer a typed credential type over a loose string-based secret
- give slots stable names such as `openai` or `azureFoundryContentUnderstanding`
- keep the credential session shape focused on what the node actually needs
- implement `test(...)` so users get useful validation feedback
