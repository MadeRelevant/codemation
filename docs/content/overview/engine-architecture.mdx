# Engine Architecture

Most app developers do not need to know every engine subsystem.

But it is useful to understand the high-level split between:

- your workflow definitions,
- the engine that plans and executes them,
- the host runtime that adds persistence, scheduling, credentials, and APIs.

## Visual overview

![Engine architecture overview](/architecture/engine-overview.svg)

## What the engine owns

At a high level, `@codemation/core` owns:

- the workflow definition model,
- planning which nodes can run,
- execution and continuation,
- policies and limits,
- runtime contracts and ports,
- snapshot-based resume behavior.

This is why you can keep writing workflow code while the runtime around it becomes more capable.

## What the engine does not own

The engine is not your full application runtime.

It does **not** directly own:

- the web UI,
- auth and user management,
- the concrete database implementation,
- Redis or BullMQ wiring,
- framework-specific API hosting.

Those are added by the host and UI packages around the engine.

## Why this matters to app developers

This architecture gives you a practical benefit:

- you define workflows at a high level,
- the engine stays focused on orchestration,
- the host can swap between local and production-grade runtime choices without changing your workflow definitions.

In other words, your workflows stay stable while the deployment model grows.

## Mental model in one sentence

The engine is the part that turns **workflow definitions + items of data** into **durable runs**; the host and UI turn that into a usable product runtime.
